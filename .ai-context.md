# AFF Inventory Management System - AI Context

## 1. Vision & Purpose (Mục tiêu)
- **Problem:** Quản lý hàng tồn kho FMCG (Sữa/Thực phẩm) gặp khó khăn trong việc theo dõi chi tiết hạn sử dụng (HSD) theo từng lô nhập, dẫn đến rủi ro hết hạn và thất thoát.
- **User:** Nhân viên kho, nhân viên Sale (quản lý vùng), kế toán và Giám đốc phân phối.
- **Core Value:** **Tính năng "Multi-batch Expiry Tracking"** (theo dõi tồn kho theo từng ngày HSD riêng biệt) và **Phân quyền dữ liệu theo Khu vực** (Region-based Permissions).
- **Các tính năng khác:** 
  - Quản lý KH/NCC & Chi tiết đơn hàng (PO/SO).
  - Dashboard báo cáo trực quan (Revenue, Top Agents, Best Sellers).
  - Đồng bộ Auth với Database Metadata.

## 2. Tech Stack (Công nghệ)
- **Frontend:** Next.js 16.2.1 (App Router), React 19.
- **State Management:** React Hooks (useState, useMemo) & Context API (`UserContext` & `GlobalContext`).
- **Backend/API:** Supabase (PostgreSQL, Auth, RLS Server Actions).
- **Unique Libs:** Lucide React (Icons), Tailwind Merge/Clsx (Style management).
- **Charts:** **Recharts** (Line, Pie, Bar charts với tooltip tương tác mượt).

## 3. Architecture & Data Flow (Kiến trúc & Luồng dữ liệu)
- **Pattern:** Next.js App Router (Folder-by-dashboard-feature). Chia rõ Client/Server logic.
- **Primary Data Flow:** [User Action] -> [Server Actions/RPC/State update] -> [Supabase DB/Re-render].
- **Data Architecture:** Chuyển dịch hoàn toàn sang **Server-side (Lazy Loading)**. 
  - Gỡ bỏ `customers`, `debtInvoices`, `saleOrders` khỏi `GlobalContext` để giải phóng RAM.
  - Sử dụng **Async Search** (Autocomplete) cho việc chọn khách hàng trong đơn hàng.
  - Chuyển logic sinh mã ID (`SO-`, `PO-`) xuống Database thông qua **RPC (`generate_next_so_id`)** để chống trùng lặp khi làm việc quy mô lớn.
- **Auth:** Supabase Auth (SSR) liên kết với bảng `ho_so_nhan_vien`. Sử dụng **JWT Claims Sync** để tối ưu hóa RLS policies.
- **Proxy:** Middleware được triển khai tại `src/proxy.ts`.
- **Global Config:** Lấy từ bảng `app_settings` (Expiry warning, VAT, ReturnExpiryMonths).
- **Audit Logs:** Tích hợp **Activity Logs** tự động ghi lại mọi hành động Thêm/Sửa/Xóa của nhân viên thông qua Server Actions.

## 4. Logic Quan trọng (Key Logic)
1.  **Xử lý Lô hàng (Batches):**
    - Khi nhập kho, hệ thống tự động gộp lô nếu trùng ngày HSD, ngược lại sẽ tạo lô mới.
    - `expDate` đại diện của sản phẩm luôn là ngày HSD gần nhất (sắp hết hạn nhất) để phục vụ cảnh báo.
    - **Nguyên tắc FIFO:** Sản phẩm xuất kho luôn ưu tiên trừ từ lô có HSD gần nhất.
2.  **Hệ thống Cảnh báo:** 
    - **Hạn dùng (Expiry):** Dựa trên `expWarningDays` cấu hình trong Cài đặt (mặc định 60-90 ngày). Nếu mốc HSD gần nhất rơi vào vùng này, cột HSD tự động in đỏ.
    - **Đặt hàng (ROP):** Cảnh báo màu cam cho sản phẩm có ngày dự kiến đặt hàng (ROP) còn lại ≤ 7 ngày.
3.  **Hoàn/Hủy đơn hàng:** Khi hủy đơn, hệ thống tự động hoàn tồn kho. Lô hàng hoàn lại được thiết lập HSD linh hoạt dựa trên `returnExpiryMonths`.
4.  **Tài chính & Dashboard:** 
    - Doanh số/Công nợ ghi nhận ngay khi đơn hàng chuyển trạng thái "Đã TT" hoặc "Chưa TT".
    - Sử dụng **Database RPC (`get_sales_aggregate_by_month`, `get_company_financials_by_month`)** để aggregate dữ liệu tại server. Điều này giải quyết triệt để lỗi tính toán sai khi số lượng đơn hàng vượt quá 1000/tháng (giới hạn của Supabase API).
    - Công nợ khách hàng được tổng hợp qua RPC `get_customer_debts_summary` thay vì xử lý mảng thủ công ở Client.
5.  **Phân quyền & Hiệu suất (Permissions & Performance):** 
    - Lọc `customers` và `orders` theo `khu_vuc_quan_ly` tại database layer.
    - Áp dụng **Server-side Search & Pagination** (Limit 50) cho tất cả danh sách nghiệp vụ chính.
6.  **Quản lý Bộ nhớ & Caching (Performance & Integrity):**
    - **Data Scale Boundaries (Ràng buộc quy mô quan trọng):** Thiết kế hệ thống tuân thủ chặt chẽ thực tế vận hành: Tối đa 5 nhà cung cấp, khoảng 20-25 sản phẩm, mỗi năm chỉ có khoảng 15 đơn nhập hàng (Purchase Orders). Các dữ liệu danh mục siêu nhỏ này được đẩy thẳng vào Client `GlobalContext` (chưa tới 5KB RAM) để tối đa hóa tốc độ UI (O(1)).
    - **Transactional Data (Dữ liệu động):** Ngược lại, Khách hàng (Customers), Đơn bán (Sale Orders), và Chi tiết Công nợ (Debt Invoices) do đặc thù mở rộng lên tới hàng vạn dòng nên BẮT BUỘC sử dụng **Server-Side Pagination** (.range, .limit, count='exact').
    - **Failsafe Limits:** Mọi câu lệnh fetch danh sách không phân trang (VD: Danh sách POs hiển thị) đều được giới hạn cứng (`.limit(300)`) - đủ để hiển thị 20 năm hoạt động nhập hàng mà không lo Memory Leak.
    - **Cache Invalidation:** Sử dụng `revalidateTag('inventory')` trong mọi Server Action thay đổi dữ liệu (Thêm/Sửa SP, Cập nhật trạng thái đơn) để đảm bảo tính Realtime của bộ nhớ đệm `unstable_cache`.

## 5. Transfer List (File ưu tiên - Đọc theo thứ tự)
*Khi chuyển giao hoặc tiếp tục dự án, Agent nên đọc các file này trước để nắm 80% logic hệ thống:*
1.  `.ai-context.md`: File này (Tổng quan & Quy chuẩn).
2.  `src/lib/store/GlobalContext.tsx`: Core State Manager liên kết Tồn kho, Đơn hàng, Công nợ, Doanh thu.
3.  `src/app/(dashboard)/inventory/actions.ts`: Server Actions (CRUD, RPC) và logic Caching (unstable_cache).
4.  `src/app/(dashboard)/settings/page.tsx`: Cấu hình hệ thống & Tab Hướng dẫn (Documentation).
5.  `src/app/(dashboard)/reports/page.tsx`: Dashboard báo cáo với UI KPI tinh gọn.
6.  `supabase_setup.sql`: Cấu trúc bảng và Trigger Database (Supabase).

## 6. Team Rules & Conventions (Quy tắc)
- **UI Aesthetic:** **Premium Minimalism**. Bo góc `2xl`, Shadow nhẹ, Layout sạch sẽ.
- **Slim Bar KPI:** Sử dụng thanh ngang tinh gọn trong header trang để hiển thị các chỉ số hành động (thay vì card lớn chiếm diện tích).
- **No Placeholders:** Không dùng ảnh rác, dùng `generate_image` để tạo demo chuyên nghiệp.
- **Default Landing Page:** `/dashboard` (đã được cấu hình trong `src/app/page.tsx` và Redirect sau login).
- **Code Style:** TypeScript strict mode, ES6+, Lambda, Shorthand logic. Diff-only edits để tiết kiệm token.
- **Communication:** Ngôn ngữ Tiếng Việt, xưng hô Anh/Em. Phản hồi ngắn gọn, đi thẳng vào giải pháp.
